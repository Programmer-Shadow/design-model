# 责任链模式（Chain of Responsibility）

## 一、核心思想

> 将请求沿一条处理器链依次传递，每个处理器决定**放行**或**拦截**。发送方不需要知道谁会处理请求，处理器之间也互不感知。

关键词：**解耦请求发送方和处理方**、**链条可动态组装**。

---

## 二、结构图

```
请求 → [ParamCheck] → [StockCheck] → [RiskCheck] → [Approval]
            │               │              │             │
          通过/拒绝       通过/拒绝      通过/拒绝      通过
            ↓               ↓              ↓             ↓
        拒绝则终止      拒绝则终止     拒绝则终止      最终结果
```

每个节点只关心自己的校验逻辑，不知道链上还有谁。

---

## 三、核心代码解析

### 3.1 抽象处理器（链条骨架）

```java
// chain/handler/OrderHandler.java

public abstract class OrderHandler {

    private OrderHandler next;  // 指向下一个处理器

    // 链式设置：a.setNext(b).setNext(c)
    public OrderHandler setNext(OrderHandler next) {
        this.next = next;
        return next;
    }

    // 处理请求：先执行自身逻辑，通过则传给下���个
    public final HandleResult handle(OrderRequest request, List<HandleResult> trace) {
        HandleResult result = doHandle(request);
        trace.add(result);  // 记录每个节点的结果

        if (result.isPassed() && next != null) {
            return next.handle(request, trace);  // 放行 → 传递
        }
        return result;  // 拒绝 或 链尾 → 终止
    }

    // 子类实现具体校验
    protected abstract HandleResult doHandle(OrderRequest request);
}
```

**关键设计**：
- `handle()` 加 `final`：子类不能改变"通过则传递、拒绝则终止"的链条行为
- `doHandle()` 是子类的扩展点：只写校验逻辑，不管链条流转
- `trace` 参数：收集整条链的执行过程，便于前端展示

### 3.2 具体处理器

```java
// chain/handler/ParamCheckHandler.java — 参数校验

public class ParamCheckHandler extends OrderHandler {
    @Override
    protected HandleResult doHandle(OrderRequest request) {
        if (request.getQuantity() <= 0) {
            return HandleResult.reject(handlerName(), "数量必须大于 0");
        }
        return HandleResult.pass(handlerName(), "参数校验通过");
    }
}
```

```java
// chain/handler/StockCheckHandler.java — 库存校验

public class StockCheckHandler extends OrderHandler {
    private static final int MAX_STOCK = 200;

    @Override
    protected HandleResult doHandle(OrderRequest request) {
        if (request.getQuantity() > MAX_STOCK) {
            return HandleResult.reject(handlerName(),
                    "库存不足，当前库存 " + MAX_STOCK);
        }
        return HandleResult.pass(handlerName(), "库存充足");
    }
}
```

```java
// chain/handler/RiskCheckHandler.java — 风控校验

public class RiskCheckHandler extends OrderHandler {
    private static final double MAX_AMOUNT = 50000;

    @Override
    protected HandleResult doHandle(OrderRequest request) {
        if (request.totalAmount() > MAX_AMOUNT) {
            return HandleResult.reject(handlerName(), "触发风控");
        }
        return HandleResult.pass(handlerName(), "风控通过");
    }
}
```

每个处理器只做一件事，职责单一。

### 3.3 组装链条（Controller）

```java
// chain/controller/ChainController.java

OrderHandler head = new ParamCheckHandler();
head.setNext(new StockCheckHandler())
    .setNext(new RiskCheckHandler())
    .setNext(new ApprovalHandler());

List<HandleResult> trace = new ArrayList<>();
HandleResult finalResult = head.handle(request, trace);
```

**链条在调用方组装**，而不是处理器自己知道下一个是谁。这意味着：
- 可以根据业务场景动态调整链条顺序
- 可以插入/移除某个处理器而不影响其他

## 四、运行效果

| 场景 | 参数 | 结果 |
|------|------|------|
| 正常订单 | MacBook Pro × 2，单价 12999 | ✅ 全链通过，Approval 批准 |
| 参数异常 | 商品为空 | ❌ 第 1 关 ParamCheck 拦截 |
| 库存不足 | 数量 300 | ❌ 第 2 关 StockCheck 拦截 |
| 风控触发 | 单价 30000 × 2 = 60000 | ❌ 第 3 关 RiskCheck 拦截 |

请求在哪一关被拦截，后面的处理器就不会执行——这就是责任链的"短路"特性。

## 五、与其他模式对比

| 维度 | 责任链 | 策略模式 | 模板方法 |
|------|--------|---------|---------|
| 执行方式 | 链条传递，可短路 | 选一个执行 | 全部步骤都执行 |
| 处理器关系 | 链式串联 | 互相替换 | 父子继承 |
| 谁决定执行 | 每个处理器自己决定 | 调用方选择 | 父类固定流程 |
| 可动态配置 | 是（插入/移除/重排） | 是（替换策略） | 否（继承固定） |

**一句话区分**：
- 策略 = "选一个算法执行"
- 模板方法 = "按固定步骤全部执行"
- 责任链 = "挨个问，谁能处理谁来，处理不了往后传"

## 六、适用场景

1. **多层校验/审批**：参数校验 → 权限校验 → 风控 → 审批（本 Demo）
2. **过滤器链**：Servlet Filter、Spring Security FilterChain
3. **日志级别**：DEBUG → INFO → WARN → ERROR，逐级上报
4. **事件冒泡**：DOM 事件从子元素向父元素传播

### 现实中的经典实现

```
Servlet Filter Chain:
  Request → EncodingFilter → AuthFilter → LogFilter → Servlet
  
Spring Security:
  Request → CsrfFilter → AuthenticationFilter → AuthorizationFilter → Controller

Netty ChannelPipeline:
  Inbound → Decoder → BusinessHandler → Encoder → Outbound
```

## 七、本 Demo 代码结构

```
src/chain/
├── model/
│   ├── OrderRequest.java        ← 请求对象
│   └── HandleResult.java        ← 处理结果（通过/拒绝 + 消息）
├── handler/
│   ├── OrderHandler.java        ← 抽象处理器（定义链条行为）
│   ├── ParamCheckHandler.java   ← 第 1 关：参数校验
│   ├── StockCheckHandler.java   ← 第 2 关：库存校验
│   ├── RiskCheckHandler.java    ← 第 3 关：风控校验
│   └── ApprovalHandler.java     ← 第 4 关：最终审批
└── controller/
    └── ChainController.java     ← 组装链条 + HTTP 入口

static/chain.html                ← 前端页面
```

**路由**：
- `GET /chain` — 前端页面
- `GET /api/order-check?product=X&quantity=N&unitPrice=N` — 执行责任链

## 八、可优化方向

1. **链条配置化**：从配置文件或数据库读取处理器列表，运行时动态组装
2. **异步链**：每个处理器异步执行，适合 IO 密集型校验（如调外部风控接口）
3. **双向链**：处理完成后反向传递结果（如 Netty 的 Inbound/Outbound）
4. **与策略组合**：`RiskCheckHandler` 内部用策略模式选择不同风控规则
