# 单例模式（Singleton Pattern）— 五种写法对比

## 一、核心思想

> 保证一个类在 JVM 中**只有一个实例**，并提供全局访问点。

解决的问题：共享资源（连接池、配置中心、线程池）不应被重复创建。

---

## 二、五种写法

### 2.1 饿汉式

```java
public class EagerSingleton {
    private static final EagerSingleton INSTANCE = new EagerSingleton();
    private EagerSingleton() {}
    public static EagerSingleton getInstance() { return INSTANCE; }
}
```

| 线程安全 | 懒加载 | 原理 |
|---------|--------|------|
| 是 | 否 | JVM 类加载机制保证 `static final` 只初始化一次 |

**优点**：最简单，没有锁开销。
**缺点**：类加载就创建实例，如果不用就浪费内存。
**适用**：实例一定会用到，或者创建成本很低。

---

### 2.2 懒汉式 + synchronized

```java
public class LazySingleton {
    private static LazySingleton instance;
    private LazySingleton() {}

    public static synchronized LazySingleton getInstance() {
        if (instance == null) {
            instance = new LazySingleton();
        }
        return instance;
    }
}
```

| 线程安全 | 懒加载 | 原理 |
|---------|--------|------|
| 是 | 是 | 整个方法加锁 |

**优点**：支持懒加载。
**缺点**：每次 `getInstance()` 都要获取锁，并发性能差。实例已创建后锁完全多余。
**结论**：理解即可，**不推荐使用**。

---

### 2.3 双重检查锁（DCL）

```java
public class DclSingleton {
    private static volatile DclSingleton instance;  // 必须 volatile
    private DclSingleton() {}

    public static DclSingleton getInstance() {
        if (instance == null) {                     // 第一次检查（无锁）
            synchronized (DclSingleton.class) {
                if (instance == null) {             // 第二次检查（有锁）
                    instance = new DclSingleton();
                }
            }
        }
        return instance;
    }
}
```

| 线程安全 | 懒加载 | 原理 |
|---------|--------|------|
| 是 | 是 | 两次 null 检查 + volatile + synchronized |

**为什么需要两次检查？**
- 第一次：避免实例已创建后还去抢锁（性能优化）
- 第二次：防止多个线程同时通过第一次检查后重复创建

**为什么必须 volatile？**

`instance = new DclSingleton()` 不是原子操作，JVM 实际执行三步：
1. 分配内存
2. 初始化对象
3. 将 `instance` 指向内存地址

如果发生**指令重排序**（1→3→2），线程 A 执行到第 3 步时，线程 B 在第一次检查发现 `instance != null`，直接返回了一个**未初始化完成的对象** → 空指针。

`volatile` 禁止重排序，保证 2 一定在 3 之前执行。

**适用**：面试高频考点，实际项目中静态内部类更简洁。

---

### 2.4 静态内部类（推荐）

```java
public class HolderSingleton {
    private HolderSingleton() {}

    private static class Holder {
        private static final HolderSingleton INSTANCE = new HolderSingleton();
    }

    public static HolderSingleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

| 线程安全 | 懒加载 | 原理 |
|---------|--------|------|
| 是 | 是 | JVM 类加载机制 |

**为什么是懒加载？**
- `Holder` 是静态内部类，**不会随外部类加载而加载**
- 只有首次调用 `getInstance()` 时，JVM 才会加载 `Holder` 类，触发 `INSTANCE` 初始化
- JVM 保证类加载过程线程安全

**优点**：兼具饿汉式的安全性和懒汉式的延迟加载，无锁，代码简洁。
**结论**：**日常首选**。

---

### 2.5 枚举（最安全）

```java
public enum EnumSingleton {
    INSTANCE;

    public String getInfo() { return "EnumSingleton"; }
}
```

| 线程安全 | 懒加载 | 原理 |
|---------|--------|------|
| 是 | 否 | JVM 枚举机制 |

**为什么最安全？**

其他四种写法都可以被破坏：
- **反射攻击**：`Constructor.setAccessible(true)` 可以绕过 `private` 构造器创建新实例
- **序列化攻击**：反序列化会创建新对象（需要额外写 `readResolve()` 防御）

枚举天然免疫这两种攻击：
- JVM 规范禁止通过反射创建枚举实例
- 枚举的序列化/反序列化由 JVM 特殊处理，保证返回同一实例

**Effective Java 推荐**："单元素的枚举类型是实现单例的最佳方法"。

## 三、五种写法对比总表

| 写法 | 线程安全 | 懒加载 | 防反射 | 防序列化 | 性能 | 推荐度 |
|------|---------|--------|--------|---------|------|--------|
| 饿汉式 | ✅ | ❌ | ❌ | ❌ | 高 | ⭐⭐⭐ |
| 懒汉式 + sync | ✅ | ✅ | ❌ | ❌ | 低 | ⭐ |
| DCL | ✅ | ✅ | ❌ | ❌ | 高 | ⭐⭐⭐ |
| 静态内部类 | ✅ | ✅ | ❌ | ❌ | 高 | ⭐⭐⭐⭐ |
| 枚举 | ✅ | ❌ | ✅ | ✅ | 高 | ⭐⭐⭐⭐⭐ |

### 如何选择？

```
不需要懒加载，追求简洁？      → 饿汉式
需要懒加载？                → 静态内部类
需要防御反射/序列化攻击？     → 枚举
面试要讲原理？              → DCL（volatile + 双重检查）
```

## 四、单例在 Spring 中

Spring 的 `@Component` / `@Service` / `@Bean` 默认都是单例（`scope=singleton`），但它**不是用上面五种方式实现的**：

```
Spring 单例 ≠ Java 单例模式

Java 单例：private 构造器 + static getInstance()，JVM 级别唯一
Spring 单例：Bean 存在 ConcurrentHashMap 里，容器级别唯一
```

Spring 用的是**注册表模式**（Registry）—— 一个 Map 存所有 Bean，`getBean()` 先查 Map，没有才创建。这比传统单例灵活（可以被测试框架替换、支持 AOP 代理等）。

## 五、常见面试问题

**Q1：DCL 为什么必须 volatile？**
> 防止指令重排序。`new` 操作分三步（分配内存 → 初始化 → 赋值引用），重排后其他线程可能拿到未初始化的对象。

**Q2：静态内部类为什么能实现懒加载？**
> JVM 不会在加载外部类时加载内部类。只有首次引用 `Holder.INSTANCE` 时才触发 `Holder` 类加载，此时才初始化实例。

**Q3：枚举单例为什么能防反射？**
> JVM 规范明确禁止通过反射创建枚举实例。`Constructor.newInstance()` 遇到枚举类型会直接抛 `IllegalArgumentException`。

**Q4：为什么不推荐懒汉式 + synchronized？**
> 实例创建后锁就多余了，但每次调用都要竞争锁。相当于为了一次初始化，让所有后续调用都付出性能代价。
