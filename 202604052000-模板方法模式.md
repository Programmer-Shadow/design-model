# 模板方法模式（Template Method Pattern）

## 一、核心思想

> 在父类中定义算法的**骨架**，将某些步骤延迟到子类实现。子类可以在不改变算法结构的前提下，重新定义算法的某些步骤。

---

## 二、结构图

```
ReportExporter（抽象类）
│
├── export(data)          ← 模板方法（final，禁止覆盖）
│   ├── filterData()      ← 钩子方法（可选覆盖）
│   ├── buildHeader()     ← 抽象方法（必须实现）
│   ├── buildBody()       ← 抽象方法（必须实现）
│   └── buildFooter()     ← 抽象方法（必须实现）
│
├── CsvReportExporter     ← 子类：CSV 格式实现
├── JsonReportExporter    ← 子类：JSON 格式实现
└── XmlReportExporter     ← 子类：XML 格式实现
```

---

## 三、关键代码解析

### 3.1 抽象基类（模板方法核心）

```java
public abstract class ReportExporter {

    // 模板方法：final 锁死流程，子类无法改变步骤顺序
    public final String export(List<ReportData> data) {
        List<ReportData> filtered = filterData(data);  // 钩子
        String header = buildHeader();                  // 抽象
        String body   = buildBody(filtered);            // 抽象
        String footer = buildFooter();                  // 抽象
        return header + body + footer;
    }

    // 钩子方法：默认透传，子类按需覆盖（如：只导出金额 > 5000 的订单）
    protected List<ReportData> filterData(List<ReportData> data) {
        return data;
    }

    protected abstract String buildHeader();
    protected abstract String buildBody(List<ReportData> data);
    protected abstract String buildFooter();
}
```

**关键点**：
- `export()` 加 `final` 是防止子类意外破坏流程顺序
- `filterData()` 是"钩子方法"，不强制实现，给子类留扩展口

### 3.2 子类实现示例（CSV）

```java
public class CsvReportExporter extends ReportExporter {

    @Override
    protected String buildHeader() {
        return "订单号,商品,数量,总价\n";
    }

    @Override
    protected String buildBody(List<ReportData> data) {
        StringBuilder sb = new StringBuilder();
        for (ReportData row : data) {
            sb.append(row.getOrderId()).append(",")
              .append(row.getProduct()).append(",")
              .append(row.getQuantity()).append(",")
              .append(row.getTotal()).append("\n");
        }
        return sb.toString();
    }

    @Override
    protected String buildFooter() { return ""; }
}
```

子类只关心"怎么格式化"，不关心"什么时候调用"。

## 四、与策略模式的对比

| 维度 | 模板方法模式 | 策略模式 |
|------|------------|---------|
| 实现方式 | 继承（父类定义骨架） | 组合（接口注入策略） |
| 复用方式 | 子类继承公共逻辑 | 对象持有策略对象 |
| 流程控制 | 父类固定流程顺序 | 调用方自由决定 |
| 扩展方式 | 新建子类 | 新建策略实现类 |
| 适用场景 | 步骤固定、细节可变 | 算法完全可替换 |

**一句话区分**：
- 模板方法 = "流程我定，细节你填"
- 策略模式 = "接口我定，算法你换"

---

## 五、适用场景

1. **多个类有相同的执行流程，只有个别步骤不同** → 提取公共流程到父类
2. **框架设计** → 框架定义流程，业务代码实现细节（如 Spring 的 `JdbcTemplate`、`AbstractList`）
3. **防止子类破坏核心逻辑** → 用 `final` 保护模板方法

### 现实中的例子

```
JdbcTemplate.execute()
  ├── getConnection()         ← 框架处理
  ├── createStatement()       ← 框架处理
  ├── doInStatement(stmt)     ← 你实现（回调 = 钩子）
  └── closeStatement()        ← 框架处理

AbstractList.sort()
  └── 调用你实现的 get() / size()
```

## 六、本 Demo 实现

**场景**：报表导出系统支持 CSV / JSON / XML 三种格式

**路由**：
- `GET /template` — 前端页面
- `GET /api/export?format=csv|json|xml` — 执行导出，返回步骤日志 + 结果

**包结构**：
```
src/template/
├── model/ReportData.java
├── dao/ReportDAO.java
├── exporter/
│   ├── ReportExporter.java        ← 抽象模板
│   ├── CsvReportExporter.java
│   ├── JsonReportExporter.java
│   └── XmlReportExporter.java
└── controller/ReportController.java
```

## 七、可优化方向

1. **钩子扩展**：`CsvReportExporter` 覆盖 `filterData()` 实现条件过滤
2. **与策略组合**：`buildBody()` 内部再注入一个格式化策略，实现两层可变性
3. **步骤回调日志**：生产环境中可在 `export()` 里埋点监控每步耗时
