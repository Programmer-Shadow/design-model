# 工厂模式（Factory Pattern）— 三种形态对比

## 一、总览

工厂模式解决的核心问题：**把对象的创建和使用分离**，调用方不需要知道 `new` 的是谁。

三种工厂是递进关系，复杂度依次递增：

```
简单工厂 → 工厂方法 → 抽象工厂
一个 switch   一种产品多个工厂   一族产品多个工厂
```

---

## 二、简单工厂（Simple Factory）

### 2.1 一句话

一个工厂类，一个静态方法，一个 switch，根据参数创建不同对象。

### 2.2 结构

```
ShapeFactory（工厂类）
├── create("circle")    → new Circle(radius)
├── create("rectangle") → new Rectangle(w, h)
└── create("triangle")  → new Triangle(base, h)
```

### 2.3 核心代码

```java
// simplefactory/factory/ShapeFactory.java

public class ShapeFactory {
    public static Shape create(String type, double param1, double param2) {
        switch (type) {
            case "circle":    return new Circle(param1);
            case "rectangle": return new Rectangle(param1, param2);
            case "triangle":  return new Triangle(param1, param2);
            default:          throw new IllegalArgumentException("未知图形类型: " + type);
        }
    }
}
```

**调用方**（Controller）只需要：
```java
Shape shape = ShapeFactory.create(type, param1, param2);
double area = shape.area();
```

调用方不关心 `Circle`、`Rectangle` 这些具体类名。

### 2.4 优缺点

| 优点 | 缺点 |
|------|------|
| 简单直接，代码少 | 新增类型要改 switch（违反开闭原则） |
| 适合类型少且稳定的场景 | 工厂类会随类型增多而膨胀 |

### 2.5 适用场景

- 类型数量少且几乎不变（如图形类型、日志级别）
- 创建逻辑简单，不值得为每种类型单独建工厂

## 三、工厂方法（Factory Method）

### 3.1 一句话

定义一个创建对象的抽象方法，由**子类**决定实例化哪个类。每种产品对应一个工厂子类。

### 3.2 结构

```
PaymentFactory（抽象基类）
├── createProcessor()      ← 工厂方法（抽象，子类实现）
└── processPayment(amount) ← 业务方法（内部调用工厂方法）

AlipayFactory     → createProcessor() → new AlipayProcessor()
WechatPayFactory  → createProcessor() → new WechatPayProcessor()
CreditCardFactory → createProcessor() → new CreditCardProcessor()
```

### 3.3 核心代码

**抽象工厂基类**：
```java
// factorymethod/factory/PaymentFactory.java

public abstract class PaymentFactory {

    // 工厂方法：子类决定创建哪种处理器
    protected abstract PaymentProcessor createProcessor();

    // 业务方法：使用工厂方法创建的对象完成支付
    public String processPayment(double amount) {
        PaymentProcessor processor = createProcessor();  // ← 调工厂方法
        return processor.pay(amount);
    }
}
```

**具体工厂**（每种支付一个子类）：
```java
// factorymethod/factory/AlipayFactory.java

public class AlipayFactory extends PaymentFactory {
    @Override
    protected PaymentProcessor createProcessor() {
        return new AlipayProcessor();  // ← 子类决定创建谁
    }
}
```

**调用方**（Controller）：
```java
PaymentFactory factory;
switch (method) {
    case "wechat":     factory = new WechatPayFactory();  break;
    case "creditcard": factory = new CreditCardFactory(); break;
    default:           factory = new AlipayFactory();
}
String result = factory.processPayment(amount);
```

### 3.4 vs 简单工厂

| 维度 | 简单工厂 | 工厂方法 |
|------|---------|---------|
| 工厂数量 | 1 个类 | N 个子类 |
| 扩展方式 | 改 switch | **新建子类**（不改已有代码） |
| 开闭原则 | 违反 | 符合 |
| 复杂度 | 低 | 中 |

**关键区别**：新增支付方式时——
- 简单工厂：打开 `ShapeFactory.java`，在 switch 里加一行 → **改已有代码**
- 工厂方法：新建 `XXXFactory.java` + `XXXProcessor.java` → **不改已有代码**

### 3.5 适用场景

- 产品类型会持续增长（如支付方式、消息通道）
- 每种产品的创建逻辑有差异（如不同初始化参数）
- 框架设计中让使用者自定义实现（如 Spring 的 `FactoryBean`）

## 四、抽象工厂（Abstract Factory）

### 4.1 一句话

提供一个创建**一族相关产品**的接口，每个具体工厂生产风格统一的一整套产品。

### 4.2 结构

```
UIFactory（抽象工厂接口）
├── createButton() → Button
├── createCard()   → Card
└── createAlert()  → Alert

LightThemeFactory
├── createButton() → LightButton
├── createCard()   → LightCard
└── createAlert()  → LightAlert

DarkThemeFactory
├── createButton() → DarkButton
├── createCard()   → DarkCard
└── createAlert()  → DarkAlert
```

### 4.3 核心代码

**抽象工厂接口**：
```java
// abstractfactory/factory/UIFactory.java

public interface UIFactory {
    Button createButton();   // 产品 A
    Card createCard();       // 产品 B
    Alert createAlert();     // 产品 C
}
```

**具体工厂**（每个主题一个工厂，产出一整族组件）：
```java
// abstractfactory/factory/DarkThemeFactory.java

public class DarkThemeFactory implements UIFactory {
    @Override
    public Button createButton() { return new DarkButton(); }

    @Override
    public Card createCard() { return new DarkCard(); }

    @Override
    public Alert createAlert() { return new DarkAlert(); }
}
```

**调用方**（Controller）：
```java
UIFactory factory = (theme.equals("dark"))
    ? new DarkThemeFactory()
    : new LightThemeFactory();

// 通过同一个工厂创建一族产品，保证风格一致
Button button = factory.createButton();
Card card     = factory.createCard();
Alert alert   = factory.createAlert();
```

### 4.4 vs 工厂方法

| 维度 | 工厂方法 | 抽象工厂 |
|------|---------|---------|
| 每个工厂产出 | **1 种**产品 | **一族**产品 |
| 解决的问题 | 一种产品的多种实现 | 多种产品的**搭配一致性** |
| 新增产品族 | — | 新建工厂实现类 |
| 新增产品种类 | — | 要改接口（所有工厂都要加方法） |

**关键区别**：抽象工厂保证的是**产品族的一致性**——
- 用 `DarkThemeFactory` 不可能产出 `LightButton`，从结构上杜绝搭配错误
- 工厂方法只管一种产品，无法保证多种产品之间的风格统一

### 4.5 适用场景

- 系统需要多套**风格/配置/平台**的组件（如 UI 主题、跨数据库适配）
- 必须保证同一族产品搭配使用，不允许混搭
- 产品族数量会扩展，但产品种类相对稳定

### 4.6 现实中的例子

```
JDBC — 抽象工厂的经典案例：
DriverManager.getConnection(url)
├── MySQLConnection    → MySQLStatement    → MySQLResultSet     ← MySQL 产品族
├── PostgreConnection  → PostgreStatement  → PostgreResultSet   ← PostgreSQL 产品族
└── OracleConnection   → OracleStatement   → OracleResultSet   ← Oracle 产品族

同一族的 Connection/Statement/ResultSet 必须配套使用，不能 MySQLConnection + OracleStatement
```

## 五、三种工厂对比总表

| 维度 | 简单工厂 | 工厂方法 | 抽象工厂 |
|------|---------|---------|---------|
| 工厂数量 | 1 个 | N 个子类 | N 个实现类 |
| 产品数量 | 1 种 | 1 种 | **一族**（多种） |
| 扩展方式 | 改 switch | 加工厂子类 | 加工厂实现类 |
| 开闭原则 | 违反 | 符合 | 新增族符合，新增种类违反 |
| 复杂度 | 低 | 中 | 高 |
| 本项目 Demo | 图形计算器 | 支付处理 | UI 主题组件 |
| 路由 | `/simple-factory` | `/factory-method` | `/abstract-factory` |

### 如何选择？

```
产品类型少且稳定？        → 简单工厂
产品类型会增长？          → 工厂方法
需要保证一族产品搭配一致？  → 抽象工厂
```

## 七、常见误区

1. **"抽象工厂比工厂方法更高级，应该优先用"** → 错。复杂度更高不代表更好，场景匹配才是关键。大部分业务用简单工厂就够了。

2. **"简单工厂不是设计模式"** → 严格说不在 GoF 23 种里，但它是最实用的，大量框架在用（Spring 的 `BeanFactory` 本质就是简单工厂的变体）。

3. **"工厂方法的 Controller 里还是有 switch 选工厂"** → 这没问题。switch 选的是**工厂**而不是**产品**，工厂内部的创建逻辑已经解耦。实际项目中这个 switch 通常会被配置文件或依赖注入替代。
